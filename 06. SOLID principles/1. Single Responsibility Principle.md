# Single Responsibility Principle

> _A class should have one, and only one, reason to change._

Each type has one job. Swap the database? Only touch `UserRepository`. Change the display format? Only touch `UserFormatter`. Nothing else changes.

---

## The Idea

Every type should do one thing. If you can't describe what a type does without using the word "and", it's doing too much.

- `UserService` — fetches users. ✅
- `UserManager` — fetches, formats, **and** saves users. ❌

---

## Example 1 — Mixing Concerns

### ❌ Violation

```swift
class UserManager {
    func fetchUser(id: String) -> User { ... }
    func saveToDatabase(user: User) { ... }
    func formatDisplayName(user: User) -> String { ... }
    func sendWelcomeEmail(to user: User) { ... }
}
```

Four unrelated jobs in one class. Change the email logic? Open this file. Change the display format? Open this file. Every change puts everything else at risk.

Validation and networking are in the same function. Need the same email check on another screen? You'd have to duplicate it — and if the rules change, you'd have to find and update every copy.

### ✅ Correct Approach

```swift
class UserService {
    func fetchUser(id: String) -> User { ... }
}

class UserRepository {
    func save(user: User) { ... }
    func delete(user: User) { ... }
}

struct UserFormatter {
    func displayName(for user: User) -> String {
        "\(user.firstName) \(user.lastName)"
    }
}

class UserNotifier {
    func sendWelcomeEmail(to user: User) { ... }
}
```

## Example 2 — View vs Logic (SwiftUI)

### ❌ Violation

```swift
struct ExpenseListView: View {
    @State private var expenses: [Expense] = []

    var body: some View {
        List(expenses) { expense in
            Text(expense.name)
        }
        .onAppear {
            // ❌ Networking logic inside a View
            expenses = ExpenseAPI.fetchAll()
        }
    }
}
```

The view is doing two jobs — rendering UI and fetching data. SwiftUI views should only describe what to show, not where the data comes from.

### ✅ Correct Approach

```swift
struct ExpenseListView: View {
    @State private var viewModel = ExpenseViewModel()

    var body: some View {
        List(viewModel.expenses) { expense in
            Text(expense.name)
        }
        .onAppear { viewModel.load() }
    }
}

@Observable
final class ExpenseViewModel {
    var expenses: [Expense] = []

    func load() {
        // Fetches expenses and updates the list
        expenses = ExpenseRepository().fetchAll()
    }
}

class ExpenseRepository {
    func fetchAll() -> [Expense] { ... }
    func save(_ expense: Expense) { ... }
}
```

- `ExpenseListView` — renders UI
- `ExpenseViewModel` — manages state
- `ExpenseRepository` — handles data

## Example 3 — Validation

### ❌ Violation

```swift
@Observable
final class RegistrationViewModel {
    var email = ""
    var password = ""

    func register() {
        // ❌ Validation and networking in the same place
        guard email.contains("@") else { return }
        guard password.count >= 8 else { return }
        AuthAPI.register(email: email, password: password)
    }
}
```

### ✅ Correct Approach

```swift
struct EmailValidator {
    func validate(_ email: String) -> Bool {
        email.contains("@") && email.contains(".")
    }
}

struct PasswordValidator {
    func validate(_ password: String) -> Bool {
        password.count >= 8
    }
}

class AuthService {
    func register(email: String, password: String) { ... }
}

@Observable
final class RegistrationViewModel {
    var email = ""
    var password = ""

    private let emailValidator = EmailValidator()
    private let passwordValidator = PasswordValidator()
    private let authService = AuthService()

    func register() {
        guard emailValidator.validate(email),
              passwordValidator.validate(password) else { return }
        authService.register(email: email, password: password)
    }
}
```

`EmailValidator` can now be reused on any screen. Each piece is testable on its own.

---

## Don't Over-Apply It

SRP is about unrelated concerns — not about size. A `UserViewModel` that loads and deletes users is fine. Both belong to the same job. Only split when things genuinely have nothing to do with each other.

---

## Key Takeaway

When you find yourself opening one file to make two completely unrelated changes — that's the sign to split.
